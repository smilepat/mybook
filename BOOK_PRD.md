# PRD: 용어가 보이면 개발이 보인다
## 기획부터 배포까지, 개발 프로세스로 배우는 120가지 용어와 어원

> **문서 버전**: v2.1
> **작성일**: 2026-03-26
> **상태**: 기획 확정 — 집필 착수 전
> **포지셔닝**: 개발 입문자·비전공 협업자를 위한 프로세스 연계 용어·어원 가이드

---

## 0. 한 줄 콘셉트

> **"용어를 가나다순이 아니라, 독자가 실제로 겪는 개발 순서대로 배운다."**

앱을 만들 때 개발자가 실제로 밟는 단계—문제 정의 → 기획 → 설계 → 구현 → 검증 → 배포 → 운영—에 따라 용어를 배치한다.  
독자는 **"지금 내가 하는 이 단계에서 왜 이 단어가 필요한지"** 를 먼저 이해한 뒤 정의와 어원을 습득한다.

---

## 1. 독자 정의 (Who)

### 1차 독자
| 유형 | 특징 | 바라는 것 |
|------|------|----------|
| **비전공 코딩 입문자** | 부트캠프·독학·AI 도움으로 개발 시작 | 회의/문서에서 나오는 말을 놓치지 않고 싶다 |
| **주니어 개발자 (0-2년)** | 코드는 쓸 수 있지만 팀 언어가 낯섦 | 시니어가 쓰는 표현을 매끄럽게 이해하고 싶다 |

### 2차 독자
| 유형 | 특징 | 바라는 것 |
|------|------|----------|
| **PM·기획자·디자이너** | 개발팀과 협업 필요 | 개발자와 오해 없이 소통하고 싶다 |
| **바이브 코더** | AI로 빠르게 앱을 만들지만 개념이 흔들림 | 무엇을 만드는지 언어로 설명하고 싶다 |

### 독자의 핵심 고통
1. 용어 뜻은 검색하면 나오지만, **왜 지금 이 단계에서 이 용어가 필요한지**를 아무도 알려주지 않는다.
2. 어원·역사를 모르면 비슷한 용어(프레임워크 vs 라이브러리, 패치 vs 핫픽스)가 계속 헷갈린다.
3. 사전식 용어집은 **맥락 없이 나열**되어 읽어도 실무에 연결되지 않는다.

## 5. 샘플 엔트리 3개 (집필 기준 제시)

### 샘플 1: PRD (2장)

**[이 단계에서 왜 등장하는가]**
앱을 만들기로 결정한 뒤 가장 먼저 해야 할 일은 무엇을 만들지를 글로 합의하는 것이다. 이때 작성하는 문서가 바로 PRD다.

**[정의]**
PRD(Product Requirements Document)는 제품이 무엇을 왜 어떻게 만들어야 하는지를 팀 전체가 합의하기 위해 작성하는 기획 문서다.

**[어원]**
- Product: 라틴어 producere(앞으로 끌어내다) — "아직 없는 것을 세상으로 끄집어내다"
- Requirement: 라틴어 requirere(다시 찾다·요청하다) — "반드시 있어야 할 것을 요구하다"
- Document: 라틴어 documentum(가르침·증거) — "가르치고 증명하는 기록"

세 단어를 합치면: *"앞으로 끌어낼 것에 반드시 있어야 할 조건을 증거로 기록한 것"*

**[현업 예문]**
1. "PRD 없이 바로 개발 들어가면 나중에 재작업 생겨요."
2. "이 기능이 PRD에 있었나요? 스코프 확인 필요합니다."

**[유사어 비교]**
| 용어 | 뉘앙스 |
|------|--------|
| PRD | 무엇을·왜를 합의하는 기획 문서 |
| Spec (상세 명세) | 어떻게를 구체화한 기술 문서 (PRD보다 상세) |
| Backlog | PRD에서 쪼개진 개별 작업 목록 |

**[오해 포인트]**
PRD는 개발자만 쓰는 문서가 아니다. PM·기획자·디자이너가 함께 작성하며, 개발 전에 모든 이해관계자가 서명(합의)해야 효력이 있다.

**[한 줄 기억법]**
> PRD = 팀 전체의 계약서. 코드보다 먼저 써야 한다.

### 샘플 2: Dependency (4장)

**[이 단계에서 왜 등장하는가]**
개발 환경을 세팅할 때 `pip install -r requirements.txt`를 처음 실행하면 수십 개의 라이브러리가 설치된다. 이것들이 전부 디펜던시(dependency)다.

**[정의]**
Dependency(의존성)는 내 코드가 제대로 실행되기 위해 반드시 설치되어 있어야 하는 외부 패키지나 라이브러리를 가리킨다.

**[어원]**
- Depend: 라틴어 dependere — de(아래에) + pendere(매달리다)
- "아래에 매달려 있다" → 스스로 서지 못하고 다른 것에 기대야 작동한다

의미 확장: 사람의 의존 관계 → 소프트웨어 패키지 간의 의존 관계

**[현업 예문]**
1. "버전 충돌이 생겼는데 디펜던시 트리 한번 확인해볼게요."
2. "이 라이브러리 없애면 의존하는 모듈이 4개나 깨져요."

**[유사어 비교]**
| 용어 | 뉘앙스 |
|------|--------|
| Dependency | 외부 코드에 대한 의존 |
| Import | 특정 모듈을 현재 파일에서 불러오는 행위 |
| Package | 배포 가능한 코드 묶음 (dependency의 실체) |

**[오해 포인트]**
디펜던시는 많을수록 기능이 풍부해지지만, 동시에 보안 취약점과 버전 충돌 위험도 커진다. `.env.example`과 마찬가지로 `requirements.txt`도 팀 전체가 공유해야 한다.

**[한 줄 기억법]**
> 내 코드가 기대어 서는 외부 사다리들.

---

### 샘플 3: Rollback (8장)

**[이 단계에서 왜 등장하는가]**
배포 후 예상 못한 오류가 발생했다. 가장 빠른 해결책은 이전 상태로 되돌리는 것이다. 이것이 롤백이다.

**[정의]**
Rollback(롤백)은 새 버전 배포 후 문제가 발생했을 때 이전에 안정적으로 작동하던 버전으로 시스템을 되돌리는 작업이다.

**[어원]**
- Roll: 라틴어 rotulus(두루마리) → 구른다·되감는다
- Back: 뒤로
- "두루마리를 되감듯이 이전 상태로 돌린다"

데이터베이스 트랜잭션에서 먼저 쓰인 뒤(이전 상태로 되돌리는 원자적 롤백), 일반 배포 롤백으로 의미가 확장됐다.

**[현업 예문]**
1. "배포한 지 10분 만에 결제 오류가 터져서 바로 롤백했어요."
2. "롤백 플랜 없이 금요일 오후에 배포하면 안 됩니다."

**[유사어 비교]**
| 용어 | 뉘앙스 |
|------|--------|
| Rollback | 이전 버전으로 전체 되돌리기 |
| Hotfix | 롤백 대신 현재 버전에서 긴급 패치 적용 |
| Revert | 특정 커밋 1개를 취소 (Git 명령어 수준) |

**[오해 포인트]**
롤백은 실패가 아니다. 배포 전략에 롤백 플랜이 포함되어 있어야 오히려 좋은 팀이다. 롤백 없이 핫픽스만 고집하다 장애를 키우는 경우가 많다.

**[한 줄 기억법]**
> 잘못 풀린 두루마리를 되감는 것. 용기 있는 후퇴.

## 부록 F. 1-2장 원고 초고 (22개 용어)

아래 원고는 실제 본문 초안으로, 각 항목을 "단계 맥락 → 정의 → 어원 → 예문 → 오해" 순으로 통일했다.

## 1장. 문제를 정의한다 (초고)

### 1) Requirement
- 단계 맥락: 만들기 전에 "반드시 있어야 하는 것"을 합의할 때 쓰는 단어.
- 정의: 제품이 목표를 달성하기 위해 충족해야 하는 조건.
- 어원: 라틴어 `requirere`(다시 찾다, 요구하다).
- 현업 예문: "이건 요청사항이 아니라 필수 requirement인가요?"
- 오해 포인트: 아이디어는 requirement가 아니다. 검증 가능한 형태여야 requirement다.

### 2) Problem Statement
- 단계 맥락: 기능 목록보다 먼저 "무슨 문제를 푸는지"를 문장으로 고정한다.
- 정의: 해결하려는 문제를 배경·대상·영향까지 포함해 명시한 문장/문단.
- 어원: problem(앞에 던져진 과제, 그리스어 계열) + statement(분명히 진술하기).
- 현업 예문: "문제정의 없이 기능부터 쓰면 산으로 갑니다."
- 오해 포인트: 문제정의는 길게 쓰는 글이 아니라 의사결정 기준이다.

### 3) User Persona
- 단계 맥락: 누가 쓰는지 모르면 우선순위가 흔들린다.
- 정의: 핵심 사용자군의 특성과 목표를 대표하는 가상 인물 모델.
- 어원: persona는 라틴어에서 "가면, 역할" 의미.
- 현업 예문: "이 기능은 페르소나 A보다 B에게 더 중요해요."
- 오해 포인트: 페르소나는 상상 캐릭터가 아니라 데이터 기반 요약이어야 한다.

### 4) Use Case
- 단계 맥락: 사용자가 어떤 상황에서 무엇을 하려는지 흐름으로 정리한다.
- 정의: 특정 목표를 달성하기 위한 사용자-시스템 상호작용 시나리오.
- 어원: use(쓰다) + case(사례).
- 현업 예문: "로그인 실패 use case도 넣어야 합니다."
- 오해 포인트: 화면 설명만으로는 use case가 아니다. 목표와 결과가 있어야 한다.

### 5) Scope
- 단계 맥락: "이번에 어디까지 할지"를 정하지 않으면 일정이 무너진다.
- 정의: 프로젝트에서 포함할 범위와 제외할 범위를 규정한 경계.
- 어원: 그리스어 `skopos`(과녁, 목표).
- 현업 예문: "이번 스프린트 scope는 검색 기능까지입니다."
- 오해 포인트: 스코프를 넓히는 건 친절이 아니라 리스크 증가일 수 있다.

### 6) Constraint
- 단계 맥락: 예산, 시간, 기술 한계를 명시해야 현실적인 계획이 나온다.
- 정의: 선택 가능한 해법을 제한하는 조건.
- 어원: 라틴어 `constringere`(함께 묶다).
- 현업 예문: "모바일 우선이라는 제약 때문에 차트를 단순화했어요."
- 오해 포인트: 제약은 핑계가 아니라 설계 입력값이다.

### 7) Assumption
- 단계 맥락: 확인되지 않았지만 현재 판단에 필요한 전제를 분리해 적는다.
- 정의: 검증 전까지 임시로 참으로 두는 가정.
- 어원: 라틴어 `assumere`(받아들이다).
- 현업 예문: "트래픽 1만 미만 가정(assumption)으로 설계했습니다."
- 오해 포인트: 가정은 숨기면 위험하다. 문서화해야 검증할 수 있다.

### 8) Stakeholder
- 단계 맥락: 영향을 받는 사람을 빠뜨리면 나중에 큰 변경이 생긴다.
- 정의: 제품 의사결정 또는 결과에 이해관계를 가진 개인/조직.
- 어원: stake(말뚝, 지분) + holder(보유자).
- 현업 예문: "법무팀도 핵심 stakeholder라 사전 검토가 필요합니다."
- 오해 포인트: 요청을 많이 하는 사람이 항상 최종 결정권자는 아니다.

### 9) Goal
- 단계 맥락: 팀이 같은 방향을 보게 하는 상위 목표를 먼저 선언한다.
- 정의: 프로젝트가 도달하려는 최종 상태.
- 어원: goal은 고대 영어 계열에서 "도착점" 의미.
- 현업 예문: "우리 goal은 가입 전환율 20% 개선입니다."
- 오해 포인트: 기능 출시 자체는 goal이 아니라 수단이다.

### 10) KPI
- 단계 맥락: 목표 달성 여부를 숫자로 확인하려고 지표를 둔다.
- 정의: Key Performance Indicator. 성과를 측정하는 핵심 지표.
- 어원: key(핵심) + performance(수행 성과) + indicator(가리키는 것).
- 현업 예문: "KPI는 DAU보다 재방문율이 더 적합합니다."
- 오해 포인트: 측정 가능한 값이 많다고 좋은 KPI가 되지는 않는다.

### 11) Non-functional Requirement
- 단계 맥락: 기능이 같아도 품질 요구가 다르면 제품 경험이 달라진다.
- 정의: 성능, 보안, 가용성처럼 "어떻게 잘 동작해야 하는지"를 규정한 요구사항.
- 어원: functional(작동하는) 앞에 non을 붙여 기능 외 요구를 구분.
- 현업 예문: "응답시간 300ms 이하는 NFR로 명시합시다."
- 오해 포인트: NFR은 옵션이 아니다. 운영 단계의 비용을 좌우한다.

## 2장. PRD를 쓴다 (초고)

### 12) PRD
- 단계 맥락: 개발 전에 팀 합의를 문서로 고정하는 핵심 문서.
- 정의: Product Requirements Document. 제품 목표, 범위, 요구사항, 성공기준을 정리한 문서.
- 어원: product(라틴어 `producere`, 앞으로 끌어내다) + requirement + document.
- 현업 예문: "PRD 승인 전엔 구현 착수하지 않습니다."
- 오해 포인트: PRD는 한 번 쓰고 끝나는 문서가 아니라 변경 이력과 함께 관리한다.

### 13) Specification (Spec)
- 단계 맥락: PRD가 방향을 정하면, spec은 구현 수준의 상세를 고정한다.
- 정의: 기능 동작, 입력/출력, 예외 처리를 기술적으로 명세한 문서.
- 어원: 라틴어 `specere`(보다)에서 파생, 자세히 들여다본다는 뜻.
- 현업 예문: "API spec에 에러코드 표를 추가해주세요."
- 오해 포인트: PRD와 spec을 혼동하면 기획/개발 경계가 흐려진다.

### 14) Acceptance Criteria
- 단계 맥락: "완료"를 판단할 공통 기준이 필요하다.
- 정의: 요구사항이 충족되었는지 검증하는 명확한 조건 목록.
- 어원: accept(받아들이다, 라틴어 `accipere`) + criteria(판단 기준).
- 현업 예문: "AC를 Given-When-Then으로 써주세요."
- 오해 포인트: 모호한 표현("빠르게", "편하게")은 acceptance criteria가 아니다.

### 15) MVP
- 단계 맥락: 처음부터 완벽 대신, 학습 가능한 최소 제품을 빠르게 낸다.
- 정의: Minimum Viable Product. 핵심 가설 검증이 가능한 최소 기능 제품.
- 어원: viable은 라틴어 `vita`(삶) 계열, "살아남을 수 있는" 의미.
- 현업 예문: "MVP에서는 소셜로그인 제외합시다."
- 오해 포인트: MVP는 대충 만든 제품이 아니라 학습 목적의 최소 실험 제품이다.

### 16) Milestone
- 단계 맥락: 긴 프로젝트를 중간 목표로 쪼개 진행률을 관리한다.
- 정의: 일정상 중요한 완료 지점.
- 어원: mile(거리) + stone(표석), 길 위의 이정표.
- 현업 예문: "이번 마일스톤은 결제 플로우 오픈입니다."
- 오해 포인트: 마일스톤은 태스크 목록이 아니라 의미 있는 결과 지점이어야 한다.

### 17) Edge Case
- 단계 맥락: 정상 흐름만 보면 출시 후 예외 상황에서 깨진다.
- 정의: 드물지만 발생 가능한 경계 조건의 사례.
- 어원: edge(가장자리) + case(사례).
- 현업 예문: "빈 문자열 입력 edge case 테스트했나요?"
- 오해 포인트: 드문 경우를 무시하면 고객 신뢰를 크게 잃을 수 있다.

### 18) Trade-off
- 단계 맥락: 시간, 품질, 비용 사이에서 동시에 최대를 얻기 어렵다.
- 정의: 한 가치를 얻기 위해 다른 가치를 일부 포기하는 의사결정.
- 어원: trade(교환) + off(내어주다).
- 현업 예문: "출시일을 지키려면 애니메이션 품질은 trade-off가 필요해요."
- 오해 포인트: 트레이드오프는 타협의 실패가 아니라 설계의 본질이다.

### 19) Prioritization
- 단계 맥락: 해야 할 일이 많을 때 무엇을 먼저 할지 결정한다.
- 정의: 중요도와 긴급도를 기준으로 작업 순서를 정하는 행위.
- 어원: prior(먼저, 라틴어).
- 현업 예문: "사용자 영향 기준으로 우선순위를 다시 매깁시다."
- 오해 포인트: 목소리 큰 사람 기준 우선순위는 장기적으로 비용이 크다.

### 20) Backlog
- 단계 맥락: 지금 당장 못 하는 일도 잃지 않도록 보관한다.
- 정의: 향후 처리할 요구사항/작업의 정렬된 대기 목록.
- 어원: back(뒤) + log(기록), 뒤에 쌓인 기록.
- 현업 예문: "이번 분기 제외 항목은 백로그로 넘기겠습니다."
- 오해 포인트: 백로그는 쓰레기통이 아니다. 정기 정리와 재평가가 필요하다.

### 21) Out of Scope
- 단계 맥락: 안 할 일을 명시해야 범위 확장이 줄어든다.
- 정의: 현재 프로젝트 범위에 포함하지 않는 항목.
- 어원: scope(과녁, 범위) 앞에 out을 붙여 범위 밖을 표시.
- 현업 예문: "관리자 대시보드는 이번 릴리스 out of scope입니다."
- 오해 포인트: out of scope는 영구 제외가 아니라 현재 제외다.

### 22) Definition of Done (DoD)
- 단계 맥락: 팀마다 완료 기준이 달라 생기는 충돌을 막는다.
- 정의: 작업이 완료되었다고 인정하기 위한 팀 공통 체크리스트.
- 어원: define(경계를 정하다) + done(끝난 상태).
- 현업 예문: "테스트 통과와 문서 업데이트가 DoD에 포함됩니다."
- 오해 포인트: 코딩 완료는 done이 아니다. 검증/문서/배포 준비까지 포함된다.

## 부록 G. 3-4장 원고 초고 (16개 용어)

아래 원고는 1-2장과 동일한 집필 규격(단계 맥락 → 정의 → 어원 → 현업 예문 → 오해 포인트)으로 작성했다.

## 3장. 구조를 설계한다 (초고)

### 23) Architecture
- 단계 맥락: 기능 코딩 전에 시스템의 큰 뼈대를 먼저 정한다.
- 정의: 컴포넌트와 데이터 흐름, 책임 분리를 포함한 시스템의 상위 구조 설계.
- 어원: 그리스어 `arkhitekton`(수석 건축가).
- 현업 예문: "트래픽이 늘어나도 버티려면 아키텍처부터 바꿔야 해요."
- 오해 포인트: 아키텍처는 다이어그램 한 장이 아니라 운영 전략까지 포함한다.

### 24) Component
- 단계 맥락: 시스템을 역할별 조각으로 나누어 유지보수를 쉽게 만든다.
- 정의: 독립된 책임과 인터페이스를 가진 재사용 가능한 기능 단위.
- 어원: 라틴어 `componere`(함께 놓다, 조합하다).
- 현업 예문: "인증 컴포넌트를 분리하면 테스트가 쉬워집니다."
- 오해 포인트: 파일을 나눴다고 컴포넌트화가 완료되는 것은 아니다.

### 25) Interface
- 단계 맥락: 모듈끼리 어떻게 대화할지 규칙을 정하는 단계에서 등장한다.
- 정의: 서로 다른 시스템/모듈이 상호작용하는 접점과 규약.
- 어원: 라틴어 `inter`(사이) + `facies`(면, 표면).
- 현업 예문: "인터페이스만 고정하면 내부 구현은 바꿔도 됩니다."
- 오해 포인트: UI만 interface가 아니다. 함수 시그니처도 인터페이스다.

### 26) Schema
- 단계 맥락: 데이터 구조를 합의하지 않으면 API와 DB가 충돌한다.
- 정의: 데이터의 형태, 제약, 관계를 정의한 구조적 청사진.
- 어원: 그리스어 `skhema`(형태, 모양).
- 현업 예문: "스키마 먼저 맞추고 파서를 구현합시다."
- 오해 포인트: 스키마는 DB 전용 개념이 아니다. JSON, 이벤트, 메시지에도 있다.

### 27) Entity
- 단계 맥락: 도메인에서 중요한 대상을 식별하고 모델링할 때 사용한다.
- 정의: 시스템에서 고유 식별자로 추적되는 핵심 객체.
- 어원: 라틴어 `entitas`(존재).
- 현업 예문: "문서(Document)와 프로젝트(Project)를 엔티티로 나눕니다."
- 오해 포인트: 화면 요소를 그대로 엔티티로 만들면 도메인 모델이 왜곡될 수 있다.

### 28) Relationship
- 단계 맥락: 엔티티를 나눈 뒤, 서로 어떤 의미로 연결되는지 정의한다.
- 정의: 엔티티 간의 연관 규칙(1:1, 1:N, N:M 등).
- 어원: 라틴어 `relatio`(되가져옴, 연결함).
- 현업 예문: "문서와 토픽의 관계는 다대다로 봐야 합니다."
- 오해 포인트: 관계를 약하게 정의하면 검색 정확도와 데이터 일관성이 함께 떨어진다.

### 29) Index
- 단계 맥락: 데이터가 늘어날 때 조회 성능을 유지하기 위해 설계 초기에 고려한다.
- 정의: 검색 속도를 높이기 위해 특정 필드에 만든 탐색 구조.
- 어원: 라틴어 `indicare`(가리키다, 지시하다).
- 현업 예문: "title 인덱스 없으면 검색이 풀스캔으로 갑니다."
- 오해 포인트: 인덱스는 많을수록 좋은 게 아니다. 쓰기 비용이 증가한다.

### 30) Migration
- 단계 맥락: 서비스 운영 중에도 데이터 구조 변경이 필요할 때 등장한다.
- 정의: 스키마/데이터 구조를 버전 관리하며 점진적으로 이전하는 작업.
- 어원: 라틴어 `migrare`(이주하다).
- 현업 예문: "이번 배포 전에 DB 마이그레이션 스크립트 먼저 실행합니다."
- 오해 포인트: 마이그레이션은 단순 ALTER 문이 아니라 롤백 계획까지 포함해야 한다.

### 31) Abstraction
- 단계 맥락: 복잡도를 줄이기 위해 공통 패턴을 상위 개념으로 묶는다.
- 정의: 불필요한 세부를 숨기고 핵심 동작만 드러내는 설계 기법.
- 어원: 라틴어 `abstrahere`(끌어내다, 떼어내다).
- 현업 예문: "스토어 레이어를 추상화해서 DB 교체 비용을 낮춥니다."
- 오해 포인트: 추상화는 만능이 아니다. 과하면 이해 비용이 늘어난다.

### 32) Pattern
- 단계 맥락: 반복되는 문제를 검증된 방식으로 재사용할 때 쓴다.
- 정의: 자주 발생하는 설계 문제에 대한 재사용 가능한 해결 템플릿.
- 어원: pattern은 라틴어/고프랑스어 계열에서 "모범 틀" 의미.
- 현업 예문: "팩토리 패턴으로 객체 생성 로직을 분리합시다."
- 오해 포인트: 패턴 적용이 목적이 되면 오히려 코드가 복잡해진다.

## 4장. 환경을 세팅한다 (초고)

### 33) Repository
- 단계 맥락: 팀 협업을 위해 코드와 문서를 한 저장소에 관리한다.
- 정의: 소스코드, 이력, 문서를 버전 단위로 보관하는 중앙 저장소.
- 어원: 라틴어 `reponere`(다시 놓다, 보관하다).
- 현업 예문: "레포 구조부터 정리하고 브랜치 전략 잡아요."
- 오해 포인트: 레포는 파일 보관함이 아니라 협업 규칙의 중심이다.

### 34) Branch
- 단계 맥락: 본선 코드(main)를 보호하면서 기능을 병렬 개발할 때 사용한다.
- 정의: 동일 저장소에서 독립적으로 작업 이력을 분기한 선.
- 어원: branch(나뭇가지)에서 유래.
- 현업 예문: "기능별 브랜치로 쪼개서 PR 올려주세요."
- 오해 포인트: 브랜치를 오래 끌면 병합 충돌 비용이 폭증한다.

### 35) Commit
- 단계 맥락: 변경사항을 의미 단위로 저장해 추적 가능하게 만든다.
- 정의: 버전 관리 시스템에 변경을 기록하는 최소 이력 단위.
- 어원: 라틴어 `committere`(함께 보내다, 맡기다).
- 현업 예문: "커밋 메시지에 왜 바꿨는지 이유를 남겨주세요."
- 오해 포인트: 큰 덩어리 커밋은 리뷰 품질과 롤백 가능성을 떨어뜨린다.

### 36) Environment Variable
- 단계 맥락: 코드와 비밀정보를 분리해 환경별 설정을 안전하게 관리한다.
- 정의: 실행 환경(OS/컨테이너)에 주입되어 애플리케이션 동작을 제어하는 키-값 설정.
- 어원: environ(둘러싸다) + variable(바뀔 수 있는 값).
- 현업 예문: "API 키는 코드에 넣지 말고 환경변수로 주입하세요."
- 오해 포인트: `.env` 파일은 편의 수단일 뿐, 비밀정보 보호 정책 자체는 별도로 필요하다.

### 37) Dependency
- 단계 맥락: 기능 구현에 필요한 외부 패키지를 선언하고 버전을 고정한다.
- 정의: 내 코드가 실행되기 위해 필요로 하는 외부 라이브러리/모듈.
- 어원: 라틴어 `dependere`(매달리다).
- 현업 예문: "의존성 충돌 때문에 실행 환경이 깨졌습니다."
- 오해 포인트: 최신 버전이 항상 정답은 아니다. 호환성 검증이 먼저다.

### 38) Virtual Environment
- 단계 맥락: 프로젝트별 패키지 버전을 분리해 충돌을 막는다.
- 정의: 시스템 Python과 분리된 독립 실행 환경.
- 어원: virtual(잠재적인, 실체를 모사하는) + environment.
- 현업 예문: "프로젝트마다 venv를 따로 써야 재현이 됩니다."
- 오해 포인트: 가상환경을 안 쓰면 다른 프로젝트 패키지가 서로 오염될 수 있다.

### 39) Configuration
- 단계 맥락: 코드 수정 없이 동작을 바꾸기 위한 설정 레이어를 만든다.
- 정의: 애플리케이션의 옵션, 경로, 플래그를 정의한 실행 설정 집합.
- 어원: 라틴어 `configurare`(함께 형태를 만들다).
- 현업 예문: "개발/운영 설정을 config로 분리합시다."
- 오해 포인트: 설정이 코드보다 복잡해지면 운영 실수가 늘어난다.

### 40) Package Manager
- 단계 맥락: 의존성 설치·업데이트·잠금(lock)을 자동화해 재현성을 높인다.
- 정의: 패키지 검색, 설치, 버전 관리, 제거를 수행하는 도구.
- 어원: package(묶음) + manager(관리자).
- 현업 예문: "requirements.txt 기준으로 패키지 매니저로 동기화하세요."
- 오해 포인트: 패키지 매니저를 써도 잠금 파일/버전 정책이 없으면 재현성이 깨진다.

## 부록 H. 5-6장 원고 초고 (25개 용어)

아래 원고는 앞선 장과 동일한 집필 규격(단계 맥락 → 정의 → 어원 → 현업 예문 → 오해 포인트)으로 작성했다.

## 5장. 기능을 구현한다 (초고)

### 41) Framework
- 단계 맥락: 기능을 빠르게 만들기 위해 공통 구조를 먼저 선택한다.
- 정의: 애플리케이션의 전체 흐름과 규칙을 제공하는 개발 뼈대.
- 어원: frame(골조) + work(구조물).
- 현업 예문: "이건 프레임워크 라이프사이클에 맞춰 구현해야 합니다."
- 오해 포인트: 프레임워크는 편의 도구가 아니라 설계 선택을 강제하기도 한다.

### 42) Library
- 단계 맥락: 특정 기능을 직접 구현하지 않고 검증된 코드를 가져다 쓸 때 등장한다.
- 정의: 필요할 때 호출해 사용하는 재사용 코드 묶음.
- 어원: 라틴어 `liber`(책, 기록)에서 파생된 library.
- 현업 예문: "차트는 라이브러리 쓰고 도메인 로직에 집중합시다."
- 오해 포인트: 라이브러리는 흐름 제어권이 호출자(내 코드)에게 남아 있다.

### 43) Compile
- 단계 맥락: 실행 전 코드를 기계가 이해할 형태로 변환한다.
- 정의: 소스 코드를 실행 가능한 형태로 변환하는 과정.
- 어원: 라틴어 `compilare`(모아 정리하다).
- 현업 예문: "컴파일 단계에서 타입 에러가 먼저 잡혔어요."
- 오해 포인트: 컴파일 성공은 프로그램 정답을 보장하지 않는다.

### 44) Runtime
- 단계 맥락: 코드가 실제 사용자 환경에서 동작할 때의 상태를 다룬다.
- 정의: 프로그램이 메모리에 올라가 실행되는 시간/환경.
- 어원: run + time.
- 현업 예문: "로컬에선 되는데 런타임에서만 에러가 납니다."
- 오해 포인트: 빌드 타임과 런타임 문제는 원인 분석 방식이 다르다.

### 45) Interpreter
- 단계 맥락: 코드를 한 줄씩 해석 실행하는 언어 환경을 설명할 때 쓰인다.
- 정의: 소스 코드를 즉시 해석해 실행하는 실행기.
- 어원: 라틴어 `interpres`(중개자, 통역자).
- 현업 예문: "인터프리터 버전 차이 때문에 동작이 달라졌습니다."
- 오해 포인트: 인터프리터 언어도 내부적으로 컴파일/최적화를 사용할 수 있다.

### 46) Variable
- 단계 맥락: 데이터 상태를 보관하고 바꿔가며 기능을 구현한다.
- 정의: 실행 중 값이 바뀔 수 있는 이름 붙은 저장 공간.
- 어원: 라틴어 `variare`(변화시키다).
- 현업 예문: "상수와 변수 범위를 분리하면 버그가 줄어요."
- 오해 포인트: 변수명 품질이 낮으면 로직 이해 시간이 급증한다.

### 47) Function
- 단계 맥락: 반복 로직을 잘라 재사용 가능한 단위로 만든다.
- 정의: 입력을 받아 처리 후 결과를 반환하는 독립 동작 단위.
- 어원: 라틴어 `functio`(수행, 실행).
- 현업 예문: "이 함수는 부작용이 없어서 테스트가 쉽습니다."
- 오해 포인트: 길고 많은 일을 하는 함수는 유지보수 비용이 크다.

### 48) Class
- 단계 맥락: 관련 데이터와 동작을 묶어 모델링할 때 등장한다.
- 정의: 객체를 생성하기 위한 설계도(속성 + 메서드).
- 어원: 라틴어 `classis`(구분된 집단).
- 현업 예문: "도메인 규칙은 클래스 내부에 캡슐화합시다."
- 오해 포인트: 클래스 사용이 항상 좋은 건 아니다. 단순 로직엔 과할 수 있다.

### 49) Instance
- 단계 맥락: 설계도(class)에서 실제 동작 객체를 만들 때 사용한다.
- 정의: 클래스에서 생성된 구체적 객체.
- 어원: 라틴어 `instantia`(앞에 서 있는 실체).
- 현업 예문: "요청마다 새 인스턴스를 만들면 비용이 큽니다."
- 오해 포인트: 인스턴스 생명주기 관리 실패는 메모리 문제로 이어진다.

### 50) Rendering
- 단계 맥락: 데이터를 사용자에게 보이는 화면으로 변환하는 단계.
- 정의: 코드/데이터를 시각적 결과로 그려내는 과정.
- 어원: 라틴어 `reddere` → 프랑스어 `rendre` → 영어 render.
- 현업 예문: "초기 렌더링 병목 때문에 첫 화면이 늦습니다."
- 오해 포인트: 렌더링은 출력 한 번이 아니라 계산·레이아웃·그리기의 연속이다.

### 51) State
- 단계 맥락: 사용자 입력과 서버 응답에 따라 앱 상태가 계속 변한다.
- 정의: 특정 시점의 데이터/화면/로직 조건 집합.
- 어원: 라틴어 `status`(서 있는 상태).
- 현업 예문: "상태 분리만 잘해도 버그 절반은 줄어듭니다."
- 오해 포인트: 상태를 여러 곳에 중복 저장하면 동기화 지옥이 온다.

### 52) Callback
- 단계 맥락: 비동기 작업 완료 후 후속 처리를 연결할 때 사용한다.
- 정의: 다른 함수에 전달되어 나중에 호출되는 함수.
- 어원: call + back.
- 현업 예문: "콜백 체인이 깊어져서 가독성이 떨어졌어요."
- 오해 포인트: 콜백 자체가 문제라기보다 구조화되지 않은 사용이 문제다.

### 53) Asynchronous
- 단계 맥락: 느린 작업을 기다리느라 앱 전체가 멈추지 않게 처리한다.
- 정의: 작업 완료 시점이 서로 맞춰지지 않아도 독립적으로 진행되는 실행 방식.
- 어원: 그리스어 a(반대) + syn(함께) + chronos(시간).
- 현업 예문: "파일 업로드는 비동기로 돌리고 진행률만 보여줍시다."
- 오해 포인트: 비동기는 곧 병렬 실행이라는 뜻이 아니다.

## 6장. 연결하고 통합한다 (초고)

### 54) API
- 단계 맥락: 모듈·서비스 사이에 기능을 안전하게 노출할 때 정의한다.
- 정의: Application Programming Interface. 시스템 간 상호작용 규약.
- 어원: interface(사이의 접점)에서 핵심 의미를 가져온 약어.
- 현업 예문: "API 계약 먼저 고정하고 프론트/백 분리 개발합시다."
- 오해 포인트: API 문서는 구현 뒤 설명서가 아니라 선행 계약이다.

### 55) SDK
- 단계 맥락: 특정 플랫폼 기능을 빠르게 붙일 때 도구 묶음을 사용한다.
- 정의: Software Development Kit. 라이브러리, 도구, 예제를 포함한 개발 패키지.
- 어원: kit(도구 세트).
- 현업 예문: "결제 SDK 버전 올리기 전에 샌드박스 테스트 필수입니다."
- 오해 포인트: SDK 업데이트는 단순 패키지 교체가 아니라 동작 변경을 동반한다.

### 56) Endpoint
- 단계 맥락: API를 호출할 구체 주소를 설계할 때 등장한다.
- 정의: 클라이언트가 요청을 보내는 API의 최종 접근 지점(URL/경로).
- 어원: end + point.
- 현업 예문: "검색 엔드포인트는 필터 파라미터를 받도록 바꿉시다."
- 오해 포인트: 엔드포인트 증가는 관리 비용 증가와 거의 비례한다.

### 57) Protocol
- 단계 맥락: 서로 다른 시스템이 같은 방식으로 통신하도록 규칙을 맞춘다.
- 정의: 데이터 형식, 전송 방식, 에러 처리 등을 정한 통신 규약.
- 어원: 그리스어 `protokollon`(첫 장에 붙이는 공적 기록).
- 현업 예문: "HTTP는 전송 프로토콜이고 JSON은 데이터 포맷입니다."
- 오해 포인트: 프로토콜과 포맷(예: JSON)을 혼동하면 설계 대화가 꼬인다.

### 58) REST
- 단계 맥락: 웹 API 설계 원칙을 단순하고 일관되게 유지하려고 선택한다.
- 정의: Representational State Transfer. 자원 중심의 HTTP 설계 스타일.
- 어원: fielding 박사 논문에서 정립된 현대 용어.
- 현업 예문: "REST 원칙상 동사는 URL보다 메서드로 표현합시다."
- 오해 포인트: REST는 표준 문법이 아니라 아키텍처 스타일이다.

### 59) Request / Response
- 단계 맥락: 클라이언트-서버 상호작용의 기본 왕복 단위.
- 정의: 요청(Request)과 응답(Response)으로 구성된 통신 모델.
- 어원: request(요청)와 respondere(응답하다, 라틴어) 계열.
- 현업 예문: "요청 스키마와 응답 스키마를 별도로 버전 관리합시다."
- 오해 포인트: 요청 성공(200)과 비즈니스 성공은 동일하지 않을 수 있다.

### 60) Authentication
- 단계 맥락: 사용자가 누구인지 확인하는 첫 관문을 설계한다.
- 정의: 시스템 접근 주체의 신원을 검증하는 절차.
- 어원: 그리스어 `authentikos`(진짜의, 정본의).
- 현업 예문: "인증은 통과했지만 권한은 없습니다."
- 오해 포인트: 인증과 인가를 같은 것으로 보면 보안 결함이 생긴다.

### 61) Authorization
- 단계 맥락: 인증된 사용자에게 무엇을 허용할지 정책을 적용한다.
- 정의: 특정 리소스/행동에 대한 접근 권한을 부여·검증하는 절차.
- 어원: 라틴어 `auctor`(권위를 부여하는 사람) 계열.
- 현업 예문: "작성자만 수정 가능한 인가 규칙이 필요합니다."
- 오해 포인트: 인가는 역할(Role)만이 아니라 소유권·상황 조건도 포함한다.

### 62) Token
- 단계 맥락: 로그인 상태와 권한 정보를 안전하게 전달하기 위해 사용한다.
- 정의: 인증/인가 상태를 표현하는 서명된 표식 데이터.
- 어원: 중세 영어 token(징표, 표식).
- 현업 예문: "토큰 만료 시간을 짧게 두고 재발급 흐름을 설계합시다."
- 오해 포인트: 토큰은 암호 그 자체가 아니며 유출 시 대리 권한이 생긴다.

### 63) Session
- 단계 맥락: 사용자 상호작용의 연속성을 서버 측에서 관리할 때 쓰인다.
- 정의: 일정 시간 동안 유지되는 사용자 연결/상태 단위.
- 어원: 라틴어 `sessio`(앉아 있음, 머무름).
- 현업 예문: "세션 저장소를 메모리에서 Redis로 옮깁니다."
- 오해 포인트: 세션 기반과 토큰 기반은 보안·확장성 트레이드오프가 다르다.

### 64) Payload
- 단계 맥락: 요청/응답에서 실제로 전달하려는 본문 데이터를 다룬다.
- 정의: 통신 메시지에서 헤더를 제외한 실제 내용물(body).
- 어원: pay + load, 원래는 운송 분야의 "실제 적재 화물".
- 현업 예문: "페이로드를 줄이면 모바일 지연이 눈에 띄게 줄어요."
- 오해 포인트: 페이로드 최적화는 압축만이 아니라 필드 설계 문제다.

### 65) Serialization
- 단계 맥락: 객체를 네트워크 전송/저장을 위해 표준 형식으로 바꾼다.
- 정의: 메모리 객체를 JSON, 바이너리 등 전송 가능한 형식으로 변환하는 과정.
- 어원: 라틴어 `series`(연속, 순서).
- 현업 예문: "직렬화 규칙이 바뀌면 하위 호환성부터 확인합시다."
- 오해 포인트: 직렬화는 단순 변환이 아니라 버전 호환 전략과 함께 다뤄야 한다.

## 7장. 검증하고 품질을 올린다 (초고)

### 66) Test
- 단계 맥락: 기능 구현 후 의도한 동작을 확인하기 위한 기본 활동.
- 정의: 소프트웨어가 기대 조건을 충족하는지 확인하는 검증 절차.
- 어원: 라틴어 `testis`(증인) 계열.
- 현업 예문: "테스트를 통과해야 다음 단계로 넘길 수 있어요."
- 오해 포인트: 테스트는 QA팀만의 일이 아니라 개발 과정 전체의 책임이다.

### 67) Unit Test
- 단계 맥락: 가장 작은 코드 단위를 빠르게 검증할 때 사용한다.
- 정의: 함수/메서드/모듈 등 최소 단위의 동작을 검증하는 테스트.
- 어원: unit(하나의 단위) + test.
- 현업 예문: "핵심 계산 로직은 유닛 테스트부터 깔아둡시다."
- 오해 포인트: 단위 테스트가 많아도 통합 오류를 막아주지는 못한다.

### 68) Integration Test
- 단계 맥락: 여러 모듈을 붙였을 때 상호작용 오류를 확인한다.
- 정의: 컴포넌트 간 연결과 데이터 흐름을 검증하는 테스트.
- 어원: 라틴어 `integrare`(완전하게 만들다).
- 현업 예문: "API와 DB를 함께 붙인 통합 테스트가 필요합니다."
- 오해 포인트: 통합 테스트 없이 배포하면 환경 차이 문제를 늦게 발견한다.

### 69) End-to-End Test (E2E)
- 단계 맥락: 사용자 시나리오 전체가 실제로 작동하는지 마지막에 확인한다.
- 정의: UI부터 백엔드까지 실제 흐름을 종단 간으로 검증하는 테스트.
- 어원: end(끝) + end(끝), 처음부터 끝까지.
- 현업 예문: "회원가입 E2E가 깨져서 릴리스가 지연됐습니다."
- 오해 포인트: E2E는 신뢰도가 높지만 느리고 유지비가 크다.

### 70) Mock
- 단계 맥락: 외부 의존성 없이 로직만 빠르게 테스트하려고 사용한다.
- 정의: 실제 객체를 대신해 동작을 흉내 내는 테스트 대역.
- 어원: 고프랑스어 `mocquer`(흉내 내다).
- 현업 예문: "결제 API는 목으로 대체하고 비즈니스 로직만 검증합시다."
- 오해 포인트: 목을 과하게 쓰면 실제 통합 문제를 놓친다.

### 71) Stub
- 단계 맥락: 고정된 응답만 필요한 단순 테스트 시나리오에서 사용한다.
- 정의: 미리 정한 값만 반환하는 단순 대역 객체/함수.
- 어원: stub(짧게 남은 조각, 그루터기).
- 현업 예문: "실패 케이스 재현용 스텁 데이터를 준비해주세요."
- 오해 포인트: 스텁은 상호작용 검증(mock)과 목적이 다르다.

### 72) Bug
- 단계 맥락: 기대 동작과 실제 결과가 다를 때 결함을 기록하고 수정한다.
- 정의: 소프트웨어의 오류 또는 결함.
- 어원: bug(벌레). 컴퓨팅 역사에서 실제 나방 사건으로 널리 알려짐.
- 현업 예문: "재현 경로가 있어야 버그를 정확히 닫을 수 있습니다."
- 오해 포인트: 버그 보고에는 원인 추정이 아니라 재현 절차가 우선이다.

### 73) Exception
- 단계 맥락: 정상 흐름에서 벗어나는 오류 상황을 제어하기 위해 다룬다.
- 정의: 실행 중 발생한 비정상 상황을 처리하기 위한 신호.
- 어원: 라틴어 `excipere`(꺼내다, 제외하다).
- 현업 예문: "예외를 삼키지 말고 컨텍스트와 함께 로깅하세요."
- 오해 포인트: 예외 처리는 숨기는 것이 아니라 복구 또는 중단 정책을 명확히 하는 일이다.

### 74) Error
- 단계 맥락: 사용자/시스템/코드 수준의 실패를 분류해 대응한다.
- 정의: 기대한 결과를 얻지 못한 상태 또는 그 신호.
- 어원: 라틴어 `errare`(헤매다, 길을 잃다).
- 현업 예문: "이 에러는 사용자 입력 오류인지 서버 오류인지 먼저 구분합시다."
- 오해 포인트: 모든 error를 동일한 심각도로 다루면 운영 우선순위가 무너진다.

### 75) Regression
- 단계 맥락: 수정 이후 기존 기능이 깨졌는지 확인하는 검증 단계에서 등장한다.
- 정의: 변경 후 과거에 되던 기능이 다시 실패하는 현상.
- 어원: 라틴어 `regredi`(뒤로 물러나다).
- 현업 예문: "결제 수정 후 로그인 회귀(regression)가 발생했습니다."
- 오해 포인트: 회귀는 사소한 버그가 아니라 신뢰도 하락의 신호다.

### 76) Coverage
- 단계 맥락: 테스트가 코드의 어느 범위를 검증하는지 정량 확인한다.
- 정의: 테스트가 실행한 코드 비율(라인, 브랜치 등).
- 어원: cover(덮다, 포괄하다).
- 현업 예문: "커버리지는 올랐지만 핵심 시나리오 누락이 있습니다."
- 오해 포인트: 높은 커버리지가 높은 품질을 보장하지는 않는다.

### 77) Code Review
- 단계 맥락: 병합 전 변경사항을 동료가 검토해 리스크를 줄인다.
- 정의: 코드 변경의 정확성, 가독성, 유지보수성을 평가하는 협업 절차.
- 어원: re(다시) + view(보다).
- 현업 예문: "리뷰 코멘트는 사람 공격이 아니라 코드 개선 제안입니다."
- 오해 포인트: 리뷰 속도만 중시하면 결함 탐지율이 급격히 떨어진다.

### 78) Refactor-safe Test
- 단계 맥락: 구조 개선(refactor) 시 동작 동일성을 보장하기 위해 준비한다.
- 정의: 내부 구현 변경에도 외부 동작이 동일함을 검증하는 테스트 세트.
- 어원: refactor + safe(안전).
- 현업 예문: "리팩터링 전에 refactor-safe 테스트부터 확보합시다."
- 오해 포인트: 리팩터링은 기능 추가가 아니므로 테스트 기준도 달라야 한다.

### 79) Quality Gate
- 단계 맥락: 빌드/배포 직전에 최소 품질 조건을 자동으로 검사한다.
- 정의: 코드 품질, 테스트, 보안 기준을 통과해야 다음 단계로 진행하는 정책.
- 어원: quality(품질) + gate(문, 관문).
- 현업 예문: "품질 게이트 실패면 배포 파이프라인이 중단됩니다."
- 오해 포인트: 게이트가 느슨하면 형식만 남고 실제 품질은 개선되지 않는다.

### 80) Reproducibility
- 단계 맥락: "내 환경에서는 된다"를 줄이기 위해 같은 결과 재현성을 확보한다.
- 정의: 동일 입력·환경에서 동일 결과를 반복적으로 얻을 수 있는 성질.
- 어원: reproduce(다시 만들어내다).
- 현업 예문: "이 버그는 재현성이 낮아서 원인 추적이 어렵습니다."
- 오해 포인트: 재현성 문제는 개인 실수가 아니라 환경 관리 체계 문제인 경우가 많다.

## 8장. 배포하고 운영한다 (초고)

### 81) Deploy
- 단계 맥락: 개발 완료 기능을 실제 사용자 환경으로 옮긴다.
- 정의: 애플리케이션을 실행 가능한 운영 환경에 배치하는 작업.
- 어원: 라틴어 `displicare`(펼치다).
- 현업 예문: "오늘 22시에 배포 창구를 열겠습니다."
- 오해 포인트: 배포는 끝이 아니라 운영의 시작이다.

### 82) Release
- 단계 맥락: 배포된 변경 중 사용자에게 공개할 버전을 관리한다.
- 정의: 제품 기능 변경을 버전 단위로 공식 공개하는 행위/결과물.
- 어원: 라틴어 `relaxare`(풀어주다).
- 현업 예문: "릴리스 노트에 변경 범위와 영향도를 명시합시다."
- 오해 포인트: deploy와 release는 같은 시점일 수도, 다를 수도 있다.

### 83) CI/CD
- 단계 맥락: 빌드, 테스트, 배포를 자동화해 릴리스 주기를 단축한다.
- 정의: Continuous Integration/Continuous Delivery(Deployment) 파이프라인.
- 어원: continuous(계속되는) + integration/delivery.
- 현업 예문: "CI에서 테스트 깨지면 main 병합을 막습니다."
- 오해 포인트: 자동화 도구만 도입하면 CI/CD가 되는 것이 아니다. 팀 규칙이 핵심이다.

### 84) Pipeline
- 단계 맥락: 작업 단계를 연결해 반복 가능한 배포 흐름을 만든다.
- 정의: 빌드-테스트-검증-배포를 순차/병렬로 수행하는 자동화 흐름.
- 어원: pipe(관) + line(연결선).
- 현업 예문: "파이프라인 병렬화를 통해 배포 시간을 절반으로 줄였습니다."
- 오해 포인트: 파이프라인은 빨라야 하지만, 실패 원인 추적 가능성도 유지해야 한다.

### 85) Container
- 단계 맥락: 환경 차이를 줄여 어디서나 같은 실행 결과를 만들려 사용한다.
- 정의: 애플리케이션과 실행 의존성을 함께 패키징한 격리 실행 단위.
- 어원: 라틴어 `continere`(담다, 유지하다).
- 현업 예문: "컨테이너 이미지에 런타임 버전을 고정합시다."
- 오해 포인트: 컨테이너는 VM과 동일 개념이 아니다.

### 86) Environment
- 단계 맥락: 개발/스테이징/운영 환경을 분리해 위험을 제어한다.
- 정의: 애플리케이션이 실행되는 인프라·설정·데이터의 조합.
- 어원: 프랑스어 environ(둘러싸다).
- 현업 예문: "운영 환경 변수와 개발 환경 변수를 분리하세요."
- 오해 포인트: 환경 간 차이가 클수록 배포 리스크가 커진다.

### 87) Staging
- 단계 맥락: 운영 배포 전 실제와 유사한 환경에서 최종 검증한다.
- 정의: 프로덕션 직전 검증을 위한 중간 배포 환경.
- 어원: stage(무대, 발판).
- 현업 예문: "스테이징에서 결제 연동 확인 후 운영 배포합니다."
- 오해 포인트: 스테이징 데이터/권한이 운영과 지나치게 다르면 검증 효과가 낮다.

### 88) Rollback
- 단계 맥락: 배포 후 장애 발생 시 빠르게 이전 안정 상태로 되돌린다.
- 정의: 최근 변경을 취소하고 이전 버전으로 복구하는 운영 절차.
- 어원: roll + back.
- 현업 예문: "5분 내 롤백 기준을 넘겨 즉시 되돌렸습니다."
- 오해 포인트: 롤백 준비 없는 배포는 사실상 복구 계획 없는 배포다.

### 89) Patch
- 단계 맥락: 기능 전체 변경 없이 결함만 빠르게 수정 배포한다.
- 정의: 기존 버전에 덧대는 소규모 수정 배포.
- 어원: patch(덧댐, 수선 조각).
- 현업 예문: "보안 패치는 정기 릴리스와 별도로 즉시 반영합니다."
- 오해 포인트: 작은 패치라도 영향 분석과 회귀 검증은 필요하다.

### 90) Hotfix
- 단계 맥락: 운영 중 치명 문제를 긴급하게 복구할 때 사용한다.
- 정의: 긴급 장애를 해결하기 위한 즉시 수정 배포.
- 어원: hot(긴급한 상황) + fix(고치다).
- 현업 예문: "핫픽스 후 원인 분석 포스트모템을 진행합시다."
- 오해 포인트: 핫픽스가 반복되면 구조적 문제를 가리고 있다는 신호다.

### 91) Incident
- 단계 맥락: 서비스 품질 저하나 장애 발생 시 대응 프로세스를 시작한다.
- 정의: 사용자 영향이 있는 운영 이상 사건.
- 어원: 라틴어 `incidere`(발생하다, 떨어지다).
- 현업 예문: "인시던트 등급을 P1로 올리고 온콜을 호출했습니다."
- 오해 포인트: 인시던트는 개인 실수 규명이 아니라 영향 최소화가 우선이다.

### 92) Postmortem
- 단계 맥락: 장애 이후 재발 방지를 위해 원인과 개선안을 기록한다.
- 정의: 사건 종료 후 타임라인·원인·재발 방지 조치를 정리한 회고 문서.
- 어원: 라틴어 post(이후) + mors(죽음).
- 현업 예문: "포스트모템에 탐지 지연 원인까지 포함해 주세요."
- 오해 포인트: 비난 중심 회고는 다음 인시던트 때 보고 문화를 망친다.

### 93) Monitoring
- 단계 맥락: 이상 징후를 조기에 발견하기 위해 지표를 지속 관찰한다.
- 정의: 시스템 상태를 메트릭/로그/트레이스로 수집·관측하는 활동.
- 어원: 라틴어 `monere`(경고하다).
- 현업 예문: "모니터링 알람 임계값을 실사용 패턴에 맞게 조정합시다."
- 오해 포인트: 알람이 많다고 좋은 모니터링이 아니다. 유의미한 신호가 중요하다.

### 94) Log
- 단계 맥락: 문제 원인 추적과 감사(audit)를 위해 실행 기록을 남긴다.
- 정의: 시스템 이벤트를 시간순으로 기록한 데이터.
- 어원: 항해 기록(logbook)에서 유래.
- 현업 예문: "요청 ID를 로그에 넣어 추적 가능하게 만듭니다."
- 오해 포인트: 로그는 많이 남기는 것보다 구조화해서 남기는 것이 중요하다.

### 95) Latency
- 단계 맥락: 사용자 체감 성능을 좌우하는 지연 시간을 관리한다.
- 정의: 요청부터 응답까지 걸리는 지연 시간.
- 어원: 라틴어 `latere`(숨어 있다)에서 파생된 latency.
- 현업 예문: "P95 레이턴시가 목표치를 넘어 캐시 전략을 조정했습니다."
- 오해 포인트: 평균 지연만 보면 꼬리 구간(P95/P99) 문제를 놓친다.

## 부록 J. 9-10장 원고 초고 (25개 용어)

아래 원고는 동일한 집필 규격(단계 맥락 → 정의 → 어원 → 현업 예문 → 오해 포인트)으로 작성했다.

## 9장. 개선하고 확장한다 (초고)

### 96) Refactoring
- 단계 맥락: 기능은 유지하면서 코드 구조를 더 읽기 좋고 안전하게 만든다.
- 정의: 외부 동작을 바꾸지 않고 내부 구조를 개선하는 작업.
- 어원: re(다시) + factor(요소로 분해하다).
- 현업 예문: "기능 추가 전에 리팩터링으로 결합도를 낮춥시다."
- 오해 포인트: 리팩터링은 기능 추가가 아니므로 성공 기준은 "동작 동일성"이다.

### 97) Technical Debt
- 단계 맥락: 빠른 출시를 위해 미룬 설계/코드 품질 이슈를 추적한다.
- 정의: 단기 선택으로 인해 미래에 상환해야 하는 유지보수 비용.
- 어원: debt(빚) 은유.
- 현업 예문: "이번 분기에 기술부채 상환 시간을 별도 배정합시다."
- 오해 포인트: 모든 오래된 코드는 기술부채가 아니다. 비용-효과로 판단해야 한다.

### 98) Legacy
- 단계 맥락: 기존 시스템을 유지하면서 점진 개선할 때 자주 등장한다.
- 정의: 현재 기준에서 오래되었지만 운영상 중요한 기존 코드/시스템.
- 어원: 라틴어 `legare`(남기다, 물려주다).
- 현업 예문: "레거시 모듈은 테스트 보강 후 단계적으로 교체합시다."
- 오해 포인트: 레거시는 나쁜 코드가 아니라 "중요하지만 맥락이 축적된 코드"다.

### 99) Scalability
- 단계 맥락: 사용량 증가에 맞춰 성능과 비용을 균형 있게 확장한다.
- 정의: 부하 증가 시 시스템이 성능 저하를 최소화하며 확장되는 능력.
- 어원: 라틴어 `scala`(사다리, 단계).
- 현업 예문: "수평 확장성을 고려해 세션 상태를 외부 저장소로 분리합니다."
- 오해 포인트: 확장성은 인프라 증설만이 아니라 데이터 모델과 아키텍처 문제다.

### 100) Performance
- 단계 맥락: 기능이 있어도 느리면 사용자 경험이 무너진다.
- 정의: 응답속도, 처리량, 자원 효율 등 시스템 실행 품질.
- 어원: 라틴어 `performare`(완전히 수행하다).
- 현업 예문: "성능 개선은 추측이 아니라 프로파일링 데이터로 진행합시다."
- 오해 포인트: 최적화는 근거 없는 미세튜닝보다 병목 식별이 먼저다.

### 101) Optimization
- 단계 맥락: 병목 지점을 찾아 제한된 자원에서 효율을 높인다.
- 정의: 목표 지표를 개선하기 위해 코드/구조/인프라를 조정하는 과정.
- 어원: 라틴어 `optimus`(가장 좋은).
- 현업 예문: "쿼리 최적화만으로 평균 응답이 40% 줄었습니다."
- 오해 포인트: 너무 이른 최적화는 개발 속도를 저하시킬 수 있다.

### 102) Throughput
- 단계 맥락: 단위 시간에 얼마나 많은 요청을 처리하는지 측정한다.
- 정의: 시스템이 일정 시간 동안 처리할 수 있는 작업량.
- 어원: through(통과) + put(넣다).
- 현업 예문: "스로틀 조정 후 처리량(throughput)이 안정화됐습니다."
- 오해 포인트: 처리량 증가가 곧 지연시간 개선을 의미하지는 않는다.

### 103) Bottleneck
- 단계 맥락: 전체 성능을 제한하는 가장 느린 지점을 찾아 개선한다.
- 정의: 시스템 전체 처리 속도를 결정하는 병목 구간.
- 어원: bottle(병) + neck(목), 좁은 통로 비유.
- 현업 예문: "DB 커넥션 풀이 병목이라 API 증설 효과가 없었습니다."
- 오해 포인트: 체감 문제의 원인이 항상 프론트엔드인 것은 아니다.

### 104) Caching
- 단계 맥락: 반복 조회가 많은 데이터의 응답을 빠르게 하기 위해 사용한다.
- 정의: 자주 쓰는 데이터를 임시 저장해 재조회 비용을 줄이는 기법.
- 어원: 프랑스어 `cacher`(숨기다).
- 현업 예문: "읽기 트래픽 높은 엔드포인트는 캐시 TTL을 적용합시다."
- 오해 포인트: 캐시는 성능을 올리지만 데이터 신선도 문제를 동반한다.

### 105) Sharding
- 단계 맥락: 단일 DB로 감당하기 어려운 데이터량을 분산 저장한다.
- 정의: 데이터를 규칙에 따라 여러 저장소로 분할하는 수평 분산 기법.
- 어원: shard(깨진 조각).
- 현업 예문: "사용자 ID 해시 기반 샤딩 전략을 검토합시다."
- 오해 포인트: 샤딩은 운영 복잡도를 크게 증가시켜 신중한 도입이 필요하다.

### 106) Observability
- 단계 맥락: 장애 원인을 빠르게 파악하기 위해 시스템 내부 상태를 추론 가능하게 만든다.
- 정의: 로그/메트릭/트레이스를 통해 시스템 동작을 이해하고 설명할 수 있는 능력.
- 어원: observe(관찰하다) + ability.
- 현업 예문: "관측성이 낮아서 장애 원인 도출에 3시간이 걸렸습니다."
- 오해 포인트: 모니터링 도구 설치만으로 관측성이 확보되지는 않는다.

### 107) Backward Compatibility
- 단계 맥락: 기존 사용자/클라이언트를 깨지 않으면서 변경을 배포한다.
- 정의: 새 버전이 이전 버전과의 동작 호환을 유지하는 성질.
- 어원: backward(뒤 방향) + compatibility(함께 동작 가능성).
- 현업 예문: "API v2를 내더라도 v1 호환기간을 최소 한 분기 유지합시다."
- 오해 포인트: 하위 호환을 무시하면 배포 속도보다 신뢰 손실이 더 크다.

### 108) Deprecation
- 단계 맥락: 기능을 즉시 제거하지 않고 단계적으로 종료할 때 사용한다.
- 정의: 더 이상 권장하지 않음을 공지하고 대체 경로를 제공하는 정책.
- 어원: 라틴어 `deprecari`(반대 청원하다, 사용을 멈추다).
- 현업 예문: "이 엔드포인트는 deprecated 처리 후 90일 뒤 종료합니다."
- 오해 포인트: deprecated는 즉시 삭제가 아니라 "종료 예고"다.

## 10장. AI 시대 개발 프로세스 (초고)

### 109) Prompt
- 단계 맥락: 모델에게 원하는 결과 형식과 제약을 명확히 전달한다.
- 정의: 모델의 출력을 유도하는 입력 지시문.
- 어원: 라틴어 `promptus`(준비된, 즉각적인).
- 현업 예문: "프롬프트에 출력 스키마를 넣으니 실패율이 줄었습니다."
- 오해 포인트: 프롬프트는 주문이 아니라 명세에 가깝다.

### 110) Token
- 단계 맥락: 모델 입력/출력 길이와 비용을 관리할 때 핵심 단위가 된다.
- 정의: 모델이 텍스트를 처리하는 최소 단위 조각.
- 어원: 중세 영어 token(표식, 징표).
- 현업 예문: "토큰 상한을 넘겨서 요청이 잘렸습니다."
- 오해 포인트: 토큰은 단어와 1:1 대응하지 않는다.

### 111) Context
- 단계 맥락: 모델이 참고할 배경 정보를 설계해 답변 정확도를 높인다.
- 정의: 모델이 현재 응답을 만들 때 참조하는 입력 정보 집합.
- 어원: 라틴어 `contextus`(함께 엮다).
- 현업 예문: "컨텍스트에 최신 정책 문서를 넣어 환각을 줄입시다."
- 오해 포인트: 컨텍스트가 많다고 항상 좋은 것이 아니라 관련성이 중요하다.

### 112) Embedding
- 단계 맥락: 텍스트 의미를 수치 벡터로 바꿔 검색·추천에 활용한다.
- 정의: 텍스트/이미지를 고차원 벡터 공간에 매핑한 의미 표현.
- 어원: embed(안에 심다).
- 현업 예문: "임베딩 업데이트 주기가 길면 검색 결과가 낡아집니다."
- 오해 포인트: 임베딩은 정답이 아니라 유사성 계산의 재료다.

### 113) Vector
- 단계 맥락: 의미 유사도 계산의 기본 수학 단위로 사용한다.
- 정의: 크기와 방향을 가진 수치 표현(LLM 맥락에선 임베딩 좌표).
- 어원: 라틴어 `vector`(운반자).
- 현업 예문: "벡터 거리 기준을 코사인으로 통일합시다."
- 오해 포인트: 벡터 DB를 도입해도 데이터 품질이 낮으면 결과는 개선되지 않는다.

### 114) RAG
- 단계 맥락: 모델 파라미터 밖의 최신 지식을 연결해 답변 신뢰도를 높인다.
- 정의: Retrieval Augmented Generation. 검색 결과를 생성 단계에 주입하는 기법.
- 어원: retrieve(되찾다) + augment(증강하다) + generation(생성).
- 현업 예문: "RAG 없이는 내부 문서 기반 QA 정확도가 낮았습니다."
- 오해 포인트: RAG는 환각을 줄이지만 완전히 제거하지는 못한다.

### 115) Agent
- 단계 맥락: 단일 답변을 넘어 도구 호출과 다단계 작업을 자동화한다.
- 정의: 목표 달성을 위해 판단·행동·도구 사용을 수행하는 AI 실행 단위.
- 어원: 라틴어 `agere`(행동하다).
- 현업 예문: "에이전트에 검색/요약 도구를 붙여 리서치 시간을 단축했습니다."
- 오해 포인트: 에이전트는 만능이 아니며 권한·범위 제한이 필수다.

### 116) Orchestration
- 단계 맥락: 여러 모델/도구/에이전트의 실행 순서를 조율한다.
- 정의: 복수 작업 단위를 일관된 흐름으로 제어하는 운영 로직.
- 어원: orchestra(연주를 조율하는 공간)에서 파생.
- 현업 예문: "오케스트레이션 레이어가 없어서 실패 복구가 어려웠습니다."
- 오해 포인트: 오케스트레이션이 과하면 단순 워크플로도 과복잡해진다.

### 117) Guardrail
- 단계 맥락: 모델 출력이 정책·보안·품질 기준을 넘지 않도록 제어한다.
- 정의: 입력/출력/행동을 제한하는 안전장치 규칙 집합.
- 어원: guard(보호) + rail(난간).
- 현업 예문: "민감정보 필터를 가드레일 1단계로 넣어야 합니다."
- 오해 포인트: 가드레일은 모델 성능 저하가 아니라 운영 신뢰를 위한 필수 요소다.

### 118) Hallucination
- 단계 맥락: 사실이 아닌 내용을 그럴듯하게 생성하는 위험을 관리한다.
- 정의: 근거 없는 정보를 사실처럼 출력하는 모델 오류 현상.
- 어원: 라틴어 `alucinari`(헛보다, 헤매다).
- 현업 예문: "중요 답변은 출처 인용 없으면 환각 가능성으로 표시합시다."
- 오해 포인트: 환각은 모델 버전 탓만이 아니라 프롬프트/컨텍스트 설계 문제이기도 하다.

### 119) Fine-tuning
- 단계 맥락: 특정 도메인 성능을 높이기 위해 모델을 추가 학습시킨다.
- 정의: 사전학습 모델을 과업 데이터로 재학습해 성능을 조정하는 과정.
- 어원: fine(정밀한) + tune(조율하다).
- 현업 예문: "파인튜닝 전에는 RAG로 목표 성능 달성 가능성부터 확인합시다."
- 오해 포인트: 파인튜닝은 비용이 크며 데이터 품질이 낮으면 성능이 오히려 나빠질 수 있다.

### 120) Evaluation
- 단계 맥락: 모델/에이전트 품질을 객관 지표로 반복 측정한다.
- 정의: 정량/정성 지표로 AI 시스템의 성능과 안정성을 검증하는 절차.
- 어원: 라틴어 `valere`(가치가 있다)에서 파생.
- 현업 예문: "평가셋이 없으면 개선인지 우연인지 구분할 수 없습니다."
- 오해 포인트: 단일 점수만 보면 실제 운영 품질을 놓친다. 시나리오 기반 평가가 필요하다.
